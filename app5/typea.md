# 地域未来交付金 デジタル実装型【TYPE A】申請完全ガイド
**（旧：デジタル実装タイプ TYPE 1）**

## 第1章 TYPE Aの基本概念と戦略的位置づけ

### 1. 制度の定義と目的
**「TYPE A（地域住民等利用推進型）」**は、地域の個性を活かしたデジタルサービスを実装し、地域課題の解決や魅力向上を図るための区分です。
本区分の最大の特徴は、システムを導入すること自体ではなく、**「地域住民等がそのサービスを利用し、効果を実感できる取組」**でなければならない点にあります。

### 2. 「TYPE 1」からの決定的な変化（意識改革）
令和7年度補正予算より、従来の「TYPE 1」は「TYPE A」へと名称および要件が移行しました。申請にあたっては、以下のパラダイムシフトを強く意識する必要があります。

*   **「実装」から「利用」へ:**
    単にシステムが稼働するだけでなく、住民が実際にスマホ等で操作し、利用するシーンが必須となりました。
*   **「行政効率化」の厳格化:**
    住民の目に触れない、行政内部の業務効率化（勤怠管理、電子決裁、RPA等）のみを目的とした事業は、明確に**交付対象外**となりました。
    ※「行政事務の効率化による所要時間の短縮」や「所要日数の削減」は、TYPE Aにおける住民への効果としては認められません。
*   **「単年度」から「継続性」へ:**
    単年度で終わる実証実験ではなく、実装計画期間（交付決定〜令和9年3月末）内に確実にサービスインし、その後も継続することが求められます。

## 第2章 予算規模と補助率

*   **補助率:** **1/2**
*   **交付上限額:** 国費 **1億円**（交付対象事業費ベースで **2億円**）
*   **事業費下限:** 交付対象事業費が **100万円以上** であること。
*   **申請上限数:**
    *   都道府県：同一都道府県で最大 **9事業**
    *   市区町村：同一市区町村で最大 **5事業**
    *   ※ただし、TYPE Aにおいて「地域間連携事業」を実施する場合、上記上限の「枠外」として＋1事業申請可能です。

## 第3章 必須要件（9つのチェックポイント）詳細解説

申請書（実施計画書）を作成する際、以下の9つの要件をすべて満たしていることを論証しなければなりません。一つでも欠けると不採択（足切り）となります。各項目について、審査員が確認するポイントと記述のコツを詳述します。

### 【要件①】地域の課題解決や魅力向上に取り組むものであること
*   **要件の詳細:**
    単に「デジタル技術を導入すること」自体を目的にしてはいけません。その導入によって、具体的にどのような地域課題を解決しようとしているのか、あるいはどのような地域の魅力を向上させようとしているのかが明確でなければなりません。
*   **審査員の確認方法:**
    *   交付対象事業の実施により、どのような地域課題の解決に取り組むものであるかが、実施計画書に明確に記載されているか。
    *   現状の課題（Before）と、事業実施後の姿（After）の因果関係が論理的か。
*   **記述のポイント:**
    *   「DXの推進」といった抽象的な言葉ではなく、「高齢者の買い物難民問題」「観光客の滞在時間の短さ」「保育所入所選考の事務負担による保護者の待機ストレス」など、具体的な痛み（ペイン）や課題を記述してください。

### 【要件②】デジタルを活用したサービスを地域・暮らしに実装する取組であること
*   **要件の詳細:**
    庁舎の中だけで完結するシステム更新や、調査研究・実証実験で終わるものではなく、地域社会や住民の生活（暮らし）の中に、デジタルサービスが実際に組み込まれる（実装される）取組でなければなりません。
*   **審査員の確認方法:**
    *   交付対象事業が地域住民等に**直接裨益（ひえき）**するものであるか。
    *   庁内のみに止まるもの（職員しか使わないシステム等）は交付対象外と判定されます。
*   **記述のポイント:**
    *   サービスが稼働した際、地域のどこで、誰がその恩恵を受けるのかを具体的にイメージできるように記述してください。

### 【要件③】地域住民等への直接的な裨益があること（最重要）
*   **要件の詳細:**
    以下のいずれかのパターンに該当し、住民に**直接的なメリット**があることが必須です。
    1.  地方公共団体が導入したデジタルサービスを、**地域住民等が利用する**こと（例：防災アプリ、母子手帳アプリ）。
    2.  地方公共団体が導入したデジタルサービスを、**職員と地域住民等が利用する**ことで、地域住民等に直接裨益がもたらされること（例：書かない窓口システム）。
*   **審査員の確認方法:**
    *   地域住民等の利用シーンが具体的に記載されているか。
    *   **【重要NG基準】** TYPE Aにおけるサービス実装の効果として、「行政事務の効率化等による所要時間の短縮」や「所要日数の削減」だけでは認められません。
*   **記述のポイント:**
    *   主語を「住民」にして記述してください。「職員が楽になる」ではなく「住民の手続き負担が減る」「住民が24時間申請できるようになる」ことを強調します。

### 【要件④】事業の成果を複数年度に渡って計測するためのKPIが設定されていること
*   **要件の詳細:**
    事業の効果を一過性のもので終わらせないため、**複数年（3カ年）**にわたって成果を計測できるKPI（重要業績評価指標）を設定する必要があります。
*   **審査員の確認方法:**
    *   実施計画期間（実装計画期間1年、運営計画期間2年）について、年度ごとのKPI設定の**根拠**や、その**計測方法**について、可能な限り具体的に記載されているか。
    *   **アウトプット指標（活動指標）:** サービスの活動量を直接測れるもの（例：アプリDL数、利用者数）。
    *   **アウトカム指標（成果指標）:** 住民への裨益を直接測れるもの（例：満足度、利用率、行動変容数）。
*   **記述のポイント:**
    *   既存事業の機能追加の場合は、**「従前のKPIに上乗せした目標値」**を設定する必要があります。現状維持の目標設定は認められません。

### 【要件⑤】実装計画期間（交付決定日から令和9年3月31日）で実装すること
*   **要件の詳細:**
    いつまでも開発を続けることはできません。交付決定日（令和8年4月1日予定）から、**令和9年（2027年）3月31日までの間**に、確実にサービスの実装（供用開始）を完了させる必要があります。
*   **審査員の確認方法:**
    *   実装計画期間内の実装時期（サービスインの時期）が明記されているか。
    *   仕様検討、入札、開発、テスト、広報といった工程が、期間内に現実的に収まっているか。
*   **記述のポイント:**
    *   システム構築が複数年度にまたがり、令和9年3月末までにサービスが開始できない計画は対象外です。

### 【要件⑥】継続的なサービスであること
*   **要件の詳細:**
    一回限りのイベントや、一時的なキャンペーンではなく、**継続的に利用可能なサービス**として地域に定着させる必要があります。
*   **審査員の確認方法:**
    *   イベントなどの単発の事業ではなく、継続的に利用可能なサービスであることが説明されているか。
    *   実装期間終了後（令和10年度以降）も、サービスを維持・運営していく方針や財源の考え方が示されているか。
*   **記述のポイント:**
    *   「今年度限りの実証実験」という書きぶりはNGです。「次年度以降も〇〇の財源で継続する」という意思を示してください。

### 【要件⑦】他の地域等で既に確立されている優良なモデル・サービス等を活用するものであること
*   **要件の詳細:**
    リスクの高い独自開発（フルスクラッチ）は推奨されません。以下のいずれかを活用することが求められます。
    1.  他の地域等で既に確立されている**優良なモデル・サービス**。
    2.  地方公共団体が関与して実証事業等を実施済みであり、実装後に一定の効果があることが認められるサービス。
*   **審査員の確認方法:**
    *   交付対象事業の参考としたサービス（他自治体の先行事例や、導入予定のパッケージソフト名など）が具体的に記載されているか。
*   **記述のポイント:**
    *   「デジタル地方創生サービスカタログ」や「モデル仕様書」に掲載されているサービスを活用すると、この要件を満たすだけでなく、加点対象にもなり有利です。

### 【要件⑧】事業を推進するための体制が確立されていること
*   **要件の詳細:**
    自治体の担当者一人で進めるのではなく、組織的なバックアップ体制が必要です。
*   **審査員の確認方法:**
    *   コンソーシアムを形成するなど、事業を推進するための体制が確立されているか。
    *   庁内の関係部局だけでなく、地域の民間事業者、団体、国、専門家などが連携する体制図になっているか。
*   **記述のポイント:**
    *   サービスの実装・運営において、誰が何を担うのか（役割分担）を明確にしてください。利用者の代表（住民や受益者団体）が参画することも推奨されています。

### 【要件⑨】交付対象事業費が100万円を上回る事業であること
*   **要件の詳細:**
    事業規模が小さすぎる（100万円以下）場合は、本交付金の対象外となります。
*   **審査員の確認方法:**
    *   本申請時において、交付対象事業費（見積額の合計）が100万円を上回っているか。
    *   地域間連携の場合、連携する地方公共団体の交付対象事業費の**合計**が100万円を上回っていれば要件を満たします。
*   **記述のポイント:**
    *   見積もりを取得する際、対象外経費（庁内用PC購入費など）を除いた「交付対象経費」だけで100万円を超えているか確認してください。

## 第4章 加点措置（優遇措置）の攻略

TYPE Aの採択を勝ち取るためには、上記の必須要件を満たすだけでなく、以下の「加点項目」をどれだけ多く取り込めるかが勝負となります。加点項目は「該当する数が多いほど高評価」となります。

### 1. デジタル地方創生モデル仕様書の活用（最重要）
デジタル庁が公開している「モデル仕様書（23類型）」に準拠したシステムを導入する場合、加点されます。
*   **対象サービス:** AIオンデマンド交通、母子健康手帳アプリ、健康ポイント、オンライン診療、保育所業務支援、授業支援、AIドリル、地域通貨、オンライン申請、住民ポータル、電子図書館、避難所運営、公開型GIS、施設予約、デジタルミュージアム、スマートロック、高校入試手続、自治会業務、検診予約、保育所入所選考、観光ポータル、保護者連絡アプリ、こどもデータ連携。
*   **必須条件:** モデル仕様書に記載されている**「必須機能」を削除せずに**実装すること（追加は可）。申請時に「活用する」と宣言し、事業終了後に「実装機能回答票」で報告が必要です。

### 2. 共通SaaS等の活用
*   **概要:** 「デジタル地方創生サービスカタログ」に掲載されているSaaS等を導入する場合。
*   **狙い:** 安価で迅速な導入、安全性担保。
*   **申請記述:** カタログ掲載製品の中から選定する（またはその予定である）ことを明記します。

### 3. DMP（デジタルマーケットプレイス）の活用
*   **概要:** デジタル庁の「DMPカタログサイト」を活用してSaaSを調達する場合。
*   **条件:** DMPサイトで製品選定を行い、「DMPソフトウェア・プラン比較表」を出力して国へ提出することが条件です。

### 4. 自治体窓口DXSaaSの利用
*   **概要:** 「書かないワンストップ窓口」をガバメントクラウド上の標準サービス（自治体窓口DXSaaS）で実施する場合。
*   **条件:** デジタル庁の「窓口BPRアドバイザー派遣事業」を活用して、業務フローの見直し（BPR）を実施済み、または実施予定であることが必須です。

### 5. マイナンバーカードの利活用
*   **概要:** サービスの一部にマイナンバーカードを活用する場合。
*   **具体例:** 図書館カードとしての利用、避難所の入退所管理、施設のスマートロック解錠、地域のポイント付与時の本人確認、オンライン申請の公的個人認証など。

### 6. スタートアップの活用
*   **概要:** サービスの提供主体が「スタートアップ企業」である場合。
*   **条件:** 創業15年以内、未上場等の要件を満たす企業が、主たるサービス提供者であるか、一部機能（認証機能など）を提供する場合。
*   **調達の工夫:** 入札においてスタートアップが参入しやすいよう、「トライアル発注制度（随意契約）」の活用や、仕様書における特許技術の指定などの工夫が推奨されています。

### 7. 地域間連携の促進
*   **概要:** 複数の自治体が連携して事業を行う場合。
*   **枠外措置:** 「定住自立圏」や「連携中枢都市圏」、または「遠隔地連携」などの場合は、申請上限数の枠外として扱われます。
*   **必須KPI:** 地域間連携を行う場合は、通常のKPIに加え、「コスト低減」「品質向上」「利用者負担軽減」のいずれかを示す「付加価値KPI」の設定が必須です。

### 8. 未採択団体への優遇
*   過去にデジタル実装タイプ（TYPE 1）等で採択実績がない団体は、優先的に採択されます。

### 9. 新たな分野への取組
*   既に採択実績がある団体でも、過去に取り組んだことのない「新しい分野」のサービス実装に取り組む場合は加点されます。分野の確認は「RAIDA」で行います。

## 第5章 申請書（実施計画書）作成の実践テクニック

今回の申請から様式が「Excel」に変更され、PowerPointによる自由記述が大幅縮小されました。Excelのセルに具体的かつ論理的に記述する能力が問われます。

### 1. 実施計画書（Excel）の入力ポイント
*   **サービス概要と課題（79〜122行目付近）:**
    統計データや住民アンケートなどの客観的根拠を用いて、「なぜ今この事業が必要か」を熱量を持って記述します。利用シーンは「〇〇地区の高齢者が、通院のためにAIバスをスマホで予約し、待ち時間なく受診できる」といった具体的なストーリーを書きます。
*   **KPI設定（22行目付近）:**
    「目標：利用率30%」とする場合、「なぜ30%なのか（近隣類似サービスの事例、住民アンケートの結果等）」という積算根拠を必ず記載します。
*   **経費内訳（341行目以降）:**
    「システム構築一式」のような大括りはNGです。「要件定義費」「アプリ開発費」「クラウド利用料」「タブレット購入費（〇台）」など、詳細に内訳を記載します。初期費用（イニシャル）と運用費用（ランニング）を明確に分けます。

### 2. 事業概要スライド（PowerPoint 1枚）
*   Excelの内容を要約し、視覚的に訴求する唯一の資料です。
*   **構成案:**
    *   左上：背景・地域課題（写真やグラフ）
    *   左下：解決策（導入するサービスの概要、システム構成図）
    *   右上：具体的な利用シーン・住民メリット（イラストやアイコン）
    *   右下：期待される効果（KPI）・将来展望

## 第6章 厳格な遵守事項（NG事例とリスク管理）

### 1. 競争入札の原則化
*   地方自治法に基づき、原則として**「一般競争入札」**による調達を実施すること。
*   随意契約を行う場合は、合理的な理由（地方自治法施行令第167条の2に該当する根拠）の説明が必須です。

### 2. 事前着手の禁止
*   交付決定日（令和8年4月1日予定）より前に、事業に着手（契約・発注）することはできません。

### 3. 対象外経費
*   **庁内業務効率化のみ:** 住民サービスに還元されない、職員の勤怠管理や電子決裁システム等。
*   **単発イベント:** 一過性のイベント開催や、継続性のない講習会等。
*   **ハードウェア偏重:** サービスの実装を伴わない、単なるPC、タブレット、Wi-Fi機器等の購入。
*   **期間外ランニングコスト:** 原則、ランニングコストは自治体負担です。実装計画期間を超える期間（例：5年一括契約の4〜5年目）は対象外です。

## 第8章 まとめ：必勝のための戦略

TYPE Aの申請書を書く上で、以下の「3つのNot」を意識してください。

1.  **「行政都合」ではないか？**
    *   職員が楽になる話ばかり書いていませんか？ 主語を「住民」に書き換えてください。
2.  **「作っただけ」で終わらないか？**
    *   アプリを作って終わりではなく、どうやって住民にダウンロードさせるか、使わせるかの「泥臭い広報・普及計画」が具体的に書かれていますか？
3.  **「独自仕様」に固執していないか？**
    *   モデル仕様書やSaaS活用など、国の標準化路線に乗ることで加点を稼いでいますか？

これらの視点を持ってExcelの各項目を埋めていくことで、採択される可能性の高い申請書になります。
